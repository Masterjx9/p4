# PP2P Protocol And Test Guide

## What Was Implemented

`pp2p.py` implements an end-to-end persistent P2P protocol with:

1. Persistent identity
- Each node has a long-lived Ed25519 key in `<state-dir>/identity_ed25519.pem`.
- `peer_id` is derived from the identity public key.

2. Trusted peer mapping
- Known peers are pinned in `<state-dir>/contacts.json` with:
  - `peer_id`
  - `public_key_b64`
  - `rendezvous` (`onion` or `direct`)

3. Signed signaling envelopes
- Signaling messages contain:
  - sender/recipient peer IDs
  - timestamp
  - nonce
  - payload (`offer`/`answer`)
  - Ed25519 signature
- Receiver verifies signature, replay nonce, and sender identity.

4. Reconnect logic
- For each peer pair, one deterministic initiator is chosen by peer_id ordering.
- If the session dies, initiator automatically re-runs negotiation.
- No human action needed after initial trust exchange.

5. Tor usage model
- `onion` mode uses Tor as rendezvous signaling only.
- Data traffic is WebRTC DataChannel.
- Tor does not need to carry high-bandwidth app payloads.

6. Minimal Tor dependency boundary
- PP2P does not call Tor C internals.
- PP2P only uses:
  - Tor process launch
  - control-port `AUTHENTICATE`
  - control-port `ADD_ONION` / `DEL_ONION`
  - SOCKS5 `CONNECT` to `.onion`
- PP2P stores onion identity itself:
  - `<state-dir>/tor/onion_v3_key_blob.txt`
  - `<state-dir>/tor/onion_service_id.txt`
- Windows-first runtime: use a native `tor.exe` built from the local source subset
  (`tor_win_min_src`), not system-installed Tor and not WSL.

7. ICE behavior
- ICE session credentials (`ufrag`/`pwd`) are auto-generated by WebRTC and carried in SDP.
- You do not manually create ICE credentials.
- STUN is default; TURN is optional and configured only if you pass TURN CLI args.

## Files

1. `pp2p.py`
- Runtime, protocol, CLI commands.

2. `direct_smoketest.py`
- Automated local validation without Tor:
  - connect
  - bidirectional chat send
  - forced drop
  - automatic reconnect

3. `onion_smoketest.py`
- Automated Tor/onion validation:
  - bootstrap two Tor-backed nodes
  - exchange onion-based rendezvous identity
  - connect, chat, force drop, reconnect

## CLI Commands

1. Initialize node state
```bash
python pp2p.py init --state-dir <path>
```

2. Print invite JSON
```bash
python pp2p.py invite --state-dir <path> --mode direct --advertise 127.0.0.1:18080
```
For onion mode, run once in onion mode first so service-id exists:
```bash
python pp2p.py invite --state-dir <path> --mode onion
```

3. Add/update contact
```bash
python pp2p.py add-contact --state-dir <path> --invite-file <invite.json>
```

4. Run node
```bash
python pp2p.py run --state-dir <path> --mode direct --signal-port 18080
```
or
```bash
python pp2p.py run --state-dir <path> --mode onion --tor-bin <path-to-tor>
```
If `--tor-bin` is omitted, PP2P auto-detects local source-built Tor binaries in this repo
(Windows prefers `.exe` builds such as `tor_win_min_src/src/app/tor.exe`).

Optional TURN example:
```bash
python pp2p.py run --state-dir <path> --mode onion --tor-bin .\tor_win_min_src\src\app\tor.exe --turn-server turn:example.org:3478?transport=udp --turn-username myuser --turn-password mypass
```

Optional TURN REST auto-credential example (no manual TURN username/password):
```bash
python pp2p.py run --state-dir <path> --mode onion --tor-bin .\tor_win_min_src\src\app\tor.exe --turn-server turn:example.org:3478?transport=udp --turn-secret YOUR_SHARED_SECRET --turn-ttl-seconds 3600 --turn-user <peer-name>
```

5. Runtime commands
- `/peers`
- `/invite`
- `/send <peer_id> <text>`
- `/drop <peer_id>`
- `/quit`

## Test: Manual Two-Peer Direct Mode

Use two terminals.

1. Init peers:
```bash
python pp2p.py init --state-dir peers/alice
python pp2p.py init --state-dir peers/bob
```

2. Exchange invites:
```bash
python pp2p.py invite --state-dir peers/alice --mode direct --advertise 127.0.0.1:18101 > alice_invite.json
python pp2p.py invite --state-dir peers/bob --mode direct --advertise 127.0.0.1:18102 > bob_invite.json
python pp2p.py add-contact --state-dir peers/alice --invite-file bob_invite.json
python pp2p.py add-contact --state-dir peers/bob --invite-file alice_invite.json
```

3. Run both:
```bash
python pp2p.py run --state-dir peers/alice --mode direct --signal-port 18101
python pp2p.py run --state-dir peers/bob --mode direct --signal-port 18102
```

4. Verify:
- Send chat from either side: `/send <peer_id> hello`
- Force disconnect: `/drop <peer_id>`
- Watch automatic reconnect within retry window.

## Test: Manual Two-Peer Onion Mode

Prereq: build local minimal Tor subset for Windows (no external Tor install required).

From the repo root in PowerShell:
```powershell
.\build_tor_subset_windows.ps1
```

Expected outputs:
1. `tor_win_min_src\src\app\tor.exe`
2. `tor_win_min_src\libtor.a`

1. Init peers:
```bash
python pp2p.py init --state-dir peers/alice
python pp2p.py init --state-dir peers/bob
```

2. First run each peer in onion mode (this creates onion key/service-id files).
You can omit port arguments; PP2P auto-picks a free local trio:
```bash
python pp2p.py run --state-dir peers/alice --mode onion --tor-bin .\tor_win_min_src\src\app\tor.exe
python pp2p.py run --state-dir peers/bob --mode onion --tor-bin .\tor_win_min_src\src\app\tor.exe
```

3. From each runtime, copy `/invite` JSON and exchange out-of-band.
If writing invite files in PowerShell, use UTF-8:
```powershell
python pp2p.py invite --state-dir peers/alice --mode onion | Out-File -Encoding utf8 alice_invite.json
```

4. Add contacts with `add-contact` on each side.

5. Restart both runtimes in onion mode.

6. Verify:
- Chat works.
- Kill one side's process and start it again.
- Existing peer reconnects automatically after both are online and findable via onion rendezvous.

## Important Behavior Note

Reconnect requires both peers to be online and reachable by their rendezvous endpoint.
If one device is fully offline, reconnect waits until it comes back.
